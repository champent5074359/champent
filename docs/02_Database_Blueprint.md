# Database Blueprint

## 1. Scope and Conventions

This document is the logical database design for BusinessOS. It is designed for PostgreSQL on Supabase, but it is **not** SQL, a migration, or production code.

### Common conventions

- Every BusinessOS-owned table uses `id uuid` as its primary key, generated by the database.
- Each primary business entity includes `created_at timestamptz` and `updated_at timestamptz`.
- Money uses `numeric(14,2)`; quantities use `numeric(14,3)`; percentages use `numeric(5,2)`.
- Master records that must remain visible in history use `is_active boolean`, rather than being deleted.
- `auth.users` is owned by Supabase Auth. BusinessOS stores its application profile in `profiles`.
- Values such as role, product type, unit, status, payment method, and movement type should be PostgreSQL enums or controlled lookup tables when migrations are created.
- All branch-level transaction tables also carry `business_id` for ownership, filtering, and row-level security. Database constraints must ensure that their branch belongs to the same business.

## 2. Multi-Business and Multi-Owner Model

BusinessOS separates a person’s identity from their access to a business:

```text
auth.users ──1:1── profiles
                     │
                     └──< business_members >── businesses ──< branches
                               │                     │             │
                               └──< branch_members >─┴─────────────┘
```

- A profile may join many businesses through `business_members`.
- A business may have many members with more than one `owner` role, so co-ownership is supported.
- A business may have many branches.
- `branch_members` optionally narrows a member’s access to specific branches. If no branch assignments exist, access is governed by their business-level role; the final rule is enforced by Supabase row-level security in a later sprint.
- One member must be marked as `is_primary_owner` for administrative contact, but this does not remove other owners’ authority.

A simple product receives one default variant. Apparel can have variants for size/color; food can have one sellable variant and one or more ingredient products.

**Relationship:** `product_variants` N—N `product_variants` through `product_recipes`.

## 4. Inventory and Stock Movement System

`inventory_balances` is a fast, current snapshot. `stock_movements` is the immutable ledger and source for audit history. Every stock-changing workflow creates one or more confirmed movement rows and updates the balance in the same database transaction.

### Stock movement types

| Movement type | Quantity sign | Business event |
| --- | --- | --- |
| `purchase_in` | Positive | Stock received from a purchase |
| `sale_out` | Negative | Product sold |
| `adjustment_in` | Positive | Stock count or correction increases stock |
| `adjustment_out` | Negative | Stock count, damage, or correction reduces stock |
| `transfer_out` | Negative | Stock leaves a source branch |
| `transfer_in` | Positive | Stock reaches a destination branch |
| `production_in` | Positive | Prepared item is produced |
| `consumption_out` | Negative | Ingredient is consumed by recipe/production |
| `return_in` | Positive | Stock returned to branch |
| `return_out` | Negative | Stock returned to supplier or customer |

### 4.1 `inventory_balances`

One current balance per stock-tracked variant per branch.

| Column | Data type | Key | Description |
| --- | --- | --- | --- |
| `id` | `uuid` | PK | Balance identifier |
| `business_id` | `uuid` | FK → `businesses.id` | Ownership boundary |
| `branch_id` | `uuid` | FK → `branches.id` | Stock-holding branch |
| `product_variant_id` | `uuid` | FK → `product_variants.id` | Stocked variant |
| `quantity_on_hand` | `numeric(14,3)` |  | Current available physical quantity |
| `quantity_reserved` | `numeric(14,3)` |  | Reserved for future orders; default `0` |
| `reorder_level` | `numeric(14,3)` |  | Low-stock alert threshold |
| `last_movement_at` | `timestamptz` |  | Latest confirmed movement time |
| `created_at` | `timestamptz` |  | Created time |
| `updated_at` | `timestamptz` |  | Last changed time |

**Constraints:** unique (`branch_id`, `product_variant_id`); branch and product variant must be in the same business.

**Relationship:** `branches` 1—N `inventory_balances`; `product_variants` 1—N `inventory_balances`.

### 4.2 `stock_movements`

Append-only stock ledger. Quantity is signed: positive adds stock, negative removes stock.

| Column | Data type | Key | Description |
| --- | --- | --- | --- |
| `id` | `uuid` | PK | Movement identifier |
| `business_id` | `uuid` | FK → `businesses.id` | Ownership boundary |
| `branch_id` | `uuid` | FK → `branches.id` | Branch whose stock changes |
| `product_variant_id` | `uuid` | FK → `product_variants.id` | Moved variant |
| `movement_type` | `stock_movement_type` |  | Type from the movement catalog above |
| `quantity_delta` | `numeric(14,3)` |  | Signed stock change; never zero |
| `unit_cost` | `numeric(14,2)` |  | Unit cost at the moment of movement |
| `occurred_at` | `timestamptz` |  | Effective movement time |
| `reference_type` | `text` |  | Source document type, e.g. `purchase_item`, `sale_item`, `stock_transfer_item` |
| `reference_id` | `uuid` |  | Source row identifier; polymorphic reference documented above |
| `stock_transfer_id` | `uuid` | FK → `stock_transfers.id`, nullable | Groups transfer-out/in pair |
| `note` | `text` |  | Reason or operational note |
| `created_by_member_id` | `uuid` | FK → `business_members.id` | Member initiating the movement |
| `created_at` | `timestamptz` |  | Ledger write time |

**Constraints:** movement type and quantity sign must agree; branch, variant, transfer, and member must belong to the same business. Confirmed ledger rows are not updated or deleted; reversal uses a new movement.

**Relationship:** `branches` 1—N `stock_movements`; `product_variants` 1—N `stock_movements`; a transfer creates two linked movements.

### 4.3 `stock_transfers`

Transfer document that links stock leaving one branch with stock received by another.

| Column | Data type | Key | Description |
| --- | --- | --- | --- |
| `id` | `uuid` | PK | Transfer identifier |
| `business_id` | `uuid` | FK → `businesses.id` | Owning business |

| `source_branch_id` | `uuid` | FK → `branches.id` | Sending branch |
| `destination_branch_id` | `uuid` | FK → `branches.id` | Receiving branch |
| `status` | `transfer_status` |  | `draft`, `sent`, `received`, `cancelled` |
| `sent_at` | `timestamptz` |  | When stock leaves source |
| `received_at` | `timestamptz` |  | When stock arrives at destination |
| `created_by_member_id` | `uuid` | FK → `business_members.id` | Creating member |
| `received_by_member_id` | `uuid` | FK → `business_members.id`, nullable | Receiving member |
| `note` | `text` |  | Optional note |
| `created_at` | `timestamptz` |  | Created time |
| `updated_at` | `timestamptz` |  | Last changed time |

**Constraints:** source and destination differ and belong to the stated business; unique (`business_id`, `transfer_number`).

**Relationship:** `stock_transfers` 1—N `stock_transfer_items`; `stock_transfers` 1—N paired `stock_movements`.

**Relationship:** `stock_transfers` 1—N `stock_transfer_items`; `product_variants` 1—N `stock_transfer_items`.

## 6. Relationship Summary

```text
profiles ──< business_members >── businesses ──< branches
              │                            │         │
              └──< branch_members >────────┘         ├──< inventory_balances >── product_variants
                                                       ├──< stock_movements >───── product_variants
businesses ──< categories ──< products ──< product_variants ──< product_barcodes
                                             │
                                             └──< product_recipes >── product_variants (ingredient)

branches ──< purchases ──< purchase_items ── product_variants
branches ──< sales ──< sale_items ────────── product_variants
branches ──< stock_transfers >── branches
businesses ──< suppliers, expenses, incomes, events, goals
```

## 7. Required Integrity and Index Direction

- Add composite indexes for `business_id` plus the most common filter columns: `branch_id`, `status`, and transaction date/time.
- Index all foreign keys and create unique indexes for business-scoped codes: branch code, SKU, purchase number, receipt number, and transfer number.
- Enforce that related business, branch, user membership, category, and product records share the same `business_id`.
- Enforce one `inventory_balances` row for each branch/variant pair.
- Enforce stock movement sign by `movement_type`; never manually overwrite a balance to fix history.
- Apply Supabase row-level security in a later implementation sprint using `business_members` and `branch_members` as the access source.
